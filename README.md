# 猫课 CRM（v1）

猫课 CRM 是一套面向「客户成交 + 咨询交付 + 管理审计」场景的业务系统，目标是把销售、咨询师、管理员三个角色的日常操作沉淀为可追踪、可复盘、可审计的数据闭环。

系统聚焦于三类核心问题：

1. 销售侧如何稳定管理客户资产、成交动作与后续跟进。
2. 咨询侧如何沉淀可检索、可复用的服务记录。
3. 管理侧如何统一监管课程状态、退款与学费结余等关键经营数据。

---

## 一、系统介绍（当前版本）

### 1.1 产品定位

当前版本（v1）定位为「首个可运行闭环版本」，已打通从客户录入到咨询交付的主路径：

- 客户录入与标签化管理
- 课程成交与课程状态流转
- 退款与撤销退款
- 赠送学费申请与审核
- 咨询日志沉淀与跨角色查看
- 关键操作审计留痕

该版本重点解决“流程能跑通、数据可追踪、口径尽量一致”的问题，适合内部业务上线验证、流程固化和团队协作训练。

### 1.2 业务闭环（主流程）

1. 销售创建客户并绑定微信号，补充标签、备注、跟进计划。  
2. 销售新增已购课程并维护课程状态（已购未上/销售标记已上等）。  
3. 如发生退款，销售在课程维度执行退款或撤销退款。  
4. 管理员在管理端处理销课、退款、赠送学费审核等关键动作。  
5. 咨询师记录咨询日志，销售与管理员可查看咨询详情用于复盘。  
6. 系统通过审计日志记录核心变更，支持回溯责任与数据来源。

### 1.3 当前版本数据口径说明（重点）

- 金额单位：**统一使用“元”**（非分）。
- 学费结余：按客户口径基于赠送学费与净消费计算（退款影响净消费）。
- 咨询次数：当前按咨询日志实际条数统计（列表与详情口径一致）。

### 1.4 技术架构

- Monorepo 结构：
  - `backend/`：FastAPI + SQLModel + Alembic
  - `frontend/`：React + TypeScript + Vite + Ant Design
- 鉴权机制：JWT（Bearer Token）
- 角色权限：`admin` / `sales` / `consultant`
- API 前缀：`/api/v1`
- 数据库：PostgreSQL（异步运行时 + 同步迁移工具链）

### 1.5 版本边界

v1 以“核心能力可用”为目标，仍存在待持续完善的模块（例如部分高级分析看板、流程自动化能力等）。当前文档以“已落地能力”为准。

## 二、管理员（Admin）

### 2.1 已有能力

- 产品管理：课程增删改查、上下架、咨询类标记维护。
- 标签管理：标签分类与标签项配置。
- 人员管理：账号创建、角色/状态管理。
- 微信号管理：微信号维护与流转管理。
- 学费与销课处理：
  - 管理销课与退款相关动作
  - 审核销售发起的赠送学费申请
  - 查看关键审计日志

### 2.2 使用指导（建议流程）

1. 初始化基础主数据：产品、标签、人员、微信号。  
2. 每日关注待处理事项：销课、退款、赠送学费审核。  
3. 对金额相关动作执行“操作后复核”：确认客户累计花费与学费结余。  
4. 定期导出/查看审计日志，用于异常排查与管理复盘。

---

## 三、销售（Sales）

### 3.1 已有能力

- 客户管理：创建、编辑、筛选、备注、下次跟进维护。
- 微信号视图：按微信号查看客户资产。
- 标签管理：客户标签增删。
- 已购课程管理：
  - 新增已购课程
  - 维护课程实付金额
  - 维护课程状态（销售可操作状态）
  - 退款与撤销退款
- 咨询相关：
  - 查看客户咨询次数
  - 进入咨询详情页查看历史日志（只读）
- 学费相关：
  - 发起赠送学费申请
  - 查看申请记录与审核状态
- 数据复盘：销售维度指标查看

### 3.2 使用指导（建议流程）

1. 先维护微信号，再录入客户，确保客户归属清晰。  
2. 成交后及时维护已购课程与实付金额，避免后续统计偏差。  
3. 发生退款时第一时间登记，确保净消费与学费结余准确。  
4. 跟进咨询交付进度时，优先通过咨询详情查看最新服务记录。  
5. 每周固定复盘客户备注与下次跟进时间，防止漏跟进。

---

## 四、咨询师（Consultant）

### 4.1 已有能力

- 咨询客户列表与客户咨询详情查看。
- 咨询日志新增与编辑（本人日志可编辑）。
- 日志筛选：按日期范围、关键词检索。
- 日志编辑体验：草稿暂存与自动恢复。

### 4.2 使用指导（建议流程）

1. 每次咨询完成后当天补录日志（日期、时长、摘要、内容）。  
2. 摘要建议结构化，便于销售和管理端快速理解。  
3. 对连续服务客户，保持日志表达风格一致，便于阶段复盘。  
4. 编辑历史日志时明确“变更原因”，减少协作歧义。

## 五、快速启动

```bash
# 1) 启动后端
cd backend
python scripts/create_db.py
python scripts/seed_full_demo.py
uvicorn app.main:app --reload

# 2) 启动前端
cd ../frontend
npm install
npm run dev
```

默认地址：

- 前端：`http://localhost:5173`
- 后端：`http://localhost:8000`
